## GitHubとは？

GitHubは、プログラムのソースコードをインターネット上で共有・管理できるサービスです。世界中の開発者が協力してソフトウェアを開発するために使われています。

詳しくは[GitHub公式ドキュメント（日本語）](https://docs.github.com/ja)をご覧ください。

### GitとGitHubの違い

- **Git（ギット）**: バージョン管理システム。自分のパソコン上でコードの変更履歴を記録・管理するツールです。
- **GitHub（ギットハブ）**: Gitを使ったプロジェクトをインターネット上で共有・管理できるサービス（Webサイト）です。

簡単に言うと、**Gitは道具、GitHubはその道具を使うための場所**というイメージです。Gitで管理しているコードをGitHubにアップロードすることで、他の人と共同開発ができるようになります。

### GitHubでできること

- **コードの共有**: 自分が書いたコードを公開できる
- **共同開発**: 複数人で同じプロジェクトに貢献できる
- **バージョン管理**: コードの変更履歴を記録・管理できる
- **Issue（問題管理）**: バグや改善点を報告・議論できる
- **Pull Request（変更提案）**: コードの変更を提案・レビューできる

## 貢献の種類

rishu-appへの貢献には、さまざまな形があります。プログラミングの経験がなくても貢献できることがたくさんあります！

### コードを書かない貢献

- **ドキュメントの改善**: 説明をわかりやすくする、誤字を修正する
- **バグ報告**: 動作がおかしいところを見つけて報告する
- **機能提案**: 「こんな機能があったらいいな」というアイデアを提案する
- **質問**: わからないことを質問することで、ドキュメント改善のヒントになる
- **レビュー**: 他の人のPull Requestを見て感想やアドバイスを伝える

### コードを書く貢献

- **バグ修正**: 報告されたバグを修正する
- **新機能の実装**: 提案された機能を実装する
- **テストの追加**: コードが正しく動作することを確認するテストを書く
- **リファクタリング**: コードをより読みやすく、保守しやすくする

**どんな小さな貢献でも大歓迎です！** まずは簡単なことから始めて、少しずつステップアップしていきましょう。

## GitHubアカウントの作成

まだGitHubアカウントを持っていない方は、以下の手順で作成してください。

### 手順

1. **GitHubのサイトにアクセス**
   - ブラウザで [https://github.com](https://github.com) を開きます

2. **サインアップボタンをクリック**
   - 右上にある「Sign up」（サインアップ）ボタンをクリックします

3. **必要事項を入力**
   - **メールアドレス**: あなたのメールアドレスを入力
   - **パスワード**: 安全なパスワードを設定
   - **ユーザー名**: 好きなユーザー名を入力（他の人が既に使っていないもの）

4. **メール認証**
   - 登録したメールアドレスに確認メールが届きます
   - メール内のリンクをクリックして認証を完了させます

5. **完了！**
   - これでGitHubアカウントが作成されました

## GitHubの基本的な画面と用語

GitHubには独特の用語がたくさんあります。ここでは、貢献に必要な基本的な用語を説明します。

### リポジトリ（Repository）

プロジェクトのコードやファイルが保管されている場所のことです。「倉庫」という意味で、略して「リポ」や「レポジトリ」とも呼ばれます。

- **リポジトリのURL**: `https://github.com/ToYama170402/rishu-app`
- このページにアクセスすると、rishu-appのコードが見られます

#### リポジトリの画面

リポジトリのページには以下のようなタブがあります：

- **Code（コード）**: ソースコードとファイルが見られます（[詳細](https://docs.github.com/ja/repositories)）
- **Issues（課題）**: バグ報告や改善提案を議論する場所（[詳細](https://docs.github.com/ja/issues)）
- **Pull requests**: コードの変更提案を確認・議論する場所（[詳細](https://docs.github.com/ja/pull-requests)）
- **Actions**: 自動テストなどが実行される場所（[詳細](https://docs.github.com/ja/actions)）
- **Projects**: プロジェクトの進捗管理（[詳細](https://docs.github.com/ja/issues/planning-and-tracking-with-projects)）
- **Wiki**: ドキュメント（[詳細](https://docs.github.com/ja/communities/documenting-your-project-with-wikis)）
- **Settings**: 設定（権限がないと見られません）

### Issue（イシュー）

バグ報告や機能の提案、質問などを投稿する場所です。GitHubでの「掲示板」のようなものだと考えてください。

詳しくは[Issue公式ドキュメント（日本語）](https://docs.github.com/ja/issues)をご覧ください。

#### Issueでできること

- **バグ/不具合報告**: 動作がおかしい箇所を報告する
- **機能提案**: 「こんな機能があったらいいな」というアイデアを提案する
- **質問**: わからないことを聞く
- **議論**: 実装方法などについて話し合う

#### Issueの見方

1. リポジトリのページで「Issues」タブをクリック
2. 既存のIssueの一覧が表示されます
3. 気になるIssueをクリックすると詳細が見られます

#### Issueの状態

- **Open（開いている）**: まだ対応中、または対応されていない
- **Closed（閉じている）**: 解決済み、または対応完了

### Fork（フォーク）

他の人のリポジトリを自分のGitHubアカウントにコピーすることです。フォークすることで、元のリポジトリに影響を与えずに自由に変更を試せます。

詳しくは[Fork公式ドキュメント（日本語）](https://docs.github.com/ja/get-started/quickstart/fork-a-repo)をご覧ください。

- 元のリポジトリ: `ToYama170402/rishu-app`
- フォーク後: `あなたのユーザー名/rishu-app`

### Clone（クローン）

GitHubにあるリポジトリを自分のパソコンにダウンロードすることです。

詳しくは[Clone公式ドキュメント（日本語）](https://docs.github.com/ja/repositories/creating-and-managing-repositories/cloning-a-repository)をご覧ください。

### Branch（ブランチ）

コードの「枝分かれ」のことです。新しい機能を追加するときなど、メインのコードから分岐して作業することで、他の作業に影響を与えずに開発できます。

詳しくは[Branch公式ドキュメント（日本語）](https://docs.github.com/ja/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/about-branches)をご覧ください。

- **main（メイン）**: 本番のコード（幹）
- **feature/○○**: 新機能を開発するブランチ（枝）
- **fix/○○**: バグを修正するブランチ（枝）

### Commit（コミット）

コードの変更を記録することです。「セーブ」のようなイメージですが、変更の履歴が全て残ります。

詳しくは[Commit公式ドキュメント（日本語）](https://docs.github.com/ja/pull-requests/committing-changes-to-your-project)をご覧ください。

### Push（プッシュ）

自分のパソコンで行った変更（コミット）を、GitHubのリポジトリにアップロードすることです。

### Pull Request（プルリクエスト、略称: PR）

自分が行った変更を、元のリポジトリに取り込んでもらうためのお願い（リクエスト）です。

詳しくは[Pull Request公式ドキュメント（日本語）](https://docs.github.com/ja/pull-requests)をご覧ください。

#### Pull Requestの流れ

1. あなたがコードを変更する
2. Pull Requestを作成して「この変更を取り込んでください」とお願いする
3. メンテナー（管理者）がコードをレビュー（確認）する
4. 問題がなければマージ（取り込み）される

### Merge（マージ）

Pull Requestで提案された変更を、元のリポジトリに統合（取り込む）ことです。

## 貢献の流れ（全体像）

初めての方は、まずこの全体の流れを把握してください。詳しい手順は後のセクションで説明します。

```
1. リポジトリをフォークする
   ↓
2. 開発環境をセットアップする
   ↓
3. Issue を確認・作成する
   ↓
4. コードを変更する
   ↓
5. 変更をコミット・プッシュする
   ↓
6. Pull Request を作成する
   ↓
7. レビューを受ける
   ↓
8. マージされる（完了！）
```

> **Note**: コードを書かない貢献（ドキュメントの改善、誤字の修正など）の場合も、基本的な流れは同じです。

## ステップ3: Issueを確認・作成する

まず、何に取り組むかを決めましょう。

### 既存のIssueから選ぶ

1. **Issueページにアクセス**
   - [https://github.com/ToYama170402/rishu-app/issues](https://github.com/ToYama170402/rishu-app/issues)

2. **ラベルを確認**
   - `good first issue`: 初心者におすすめのIssue
   - `help wanted`: 協力者を募集しているIssue
   - `bug`: バグ報告
   - `enhancement`: 新機能の提案

3. **Issueを選ぶ**
   - 興味のあるIssueをクリックして内容を読む
   - まだ誰も取り組んでいないことを確認
   - 「このIssueに取り組みます」とコメントを残すと良いでしょう

### 新しいIssueを作成する

バグを見つけたり、新しい機能を提案したい場合は、Issueを作成します。

1. **New issueボタンをクリック**
   - Issueページの右上にある「New issue」ボタンをクリック

2. **テンプレートを選択**
   - **バグレポート**: バグを報告する場合
   - **機能要望**: 新機能を提案する場合
   - **質問/疑問**: 質問がある場合

3. **テンプレートに従って記入**
   - **タイトル**: わかりやすい見出しを付ける
   - **本文**: テンプレートに用意されている項目をすべて埋める
     - テンプレートは、必要な情報を漏れなく伝えるために用意されています
     - 各項目を丁寧に記入することで、問題の解決や機能の実装がスムーズになります
   
   **Issueを書くときのポイント:**
   - できるだけ具体的に書く（「動かない」よりも「ログインボタンを押すとエラーが出る」）
   - 再現手順を書く（どうすればその問題が起きるか）
   - スクリーンショットがあれば添付する
   - 環境情報を書く（OS、ブラウザなど）

4. **Submit new issue をクリック**
   - これでIssueが作成されます

## ステップ2: 開発環境をセットアップする

コードを編集するための環境を整えます。

### 2-1. 必要なツールのインストール

#### Git のインストール

Gitは、コードのバージョン管理をするためのツールです。詳しくは[Git公式ドキュメント（日本語）](https://git-scm.com/book/ja/v2)をご覧ください。

**Windowsの場合:**
1. [https://git-scm.com/download/win](https://git-scm.com/download/win) にアクセス
2. インストーラーをダウンロード
3. インストーラーを実行（基本的にデフォルト設定でOK）

**Macの場合:**
1. ターミナルを開く（「アプリケーション」→「ユーティリティ」→「ターミナル」）
2. 以下のコマンドを入力（Homebrewがインストールされていない場合は、先にHomebrewをインストール）
   ```bash
   brew install git
   ```
   
   Homebrewのインストール方法: [https://brew.sh/ja/](https://brew.sh/ja/)

**Linuxの場合:**
- Ubuntu/Debian:
  ```bash
  sudo apt update
  sudo apt install git
  ```
- Fedora:
  ```bash
  sudo dnf install git
  ```
- Arch Linux:
  ```bash
  sudo pacman -S git
  ```

#### Visual Studio Code (VS Code) のインストール

VS Codeは、コードを書くための高機能なソフトウェア（テキストエディタ）です。詳しくは[VS Code公式ドキュメント（日本語）](https://code.visualstudio.com/docs)をご覧ください。

1. [https://code.visualstudio.com/](https://code.visualstudio.com/) にアクセス
2. お使いのOSに合わせたバージョンをダウンロード
3. インストーラーを実行

#### Docker のインストール

Dockerは、開発環境を簡単に構築するためのツールです。詳しくは[Docker公式ドキュメント](https://docs.docker.com/)をご覧ください。

**Windowsの場合:**
1. [Docker Desktop for Windows](https://www.docker.com/products/docker-desktop) をダウンロード
2. インストーラーを実行
3. 再起動が必要な場合は再起動

**Macの場合:**
1. [Docker Desktop for Mac](https://www.docker.com/products/docker-desktop) をダウンロード
2. インストーラーを実行

**Linuxの場合:**
- 各ディストリビューションの公式ドキュメントに従ってインストール
- Ubuntu: [https://docs.docker.com/engine/install/ubuntu/](https://docs.docker.com/engine/install/ubuntu/)
- Arch Linux: [https://wiki.archlinux.jp/index.php/Docker](https://wiki.archlinux.jp/index.php/Docker)

#### VS Code の拡張機能をインストール

VS Codeに「Dev Containers」拡張機能を追加します。

**Dev Containersとは？**
Dev Containersは、Dockerを使って開発環境を自動的に構築する機能です。これにより、プロジェクトごとに異なる設定やツールを簡単に切り替えられます。詳しくは[Dev Containers公式ドキュメント](https://code.visualstudio.com/docs/devcontainers/containers)をご覧ください。

**インストール手順:**
1. VS Codeを起動
2. 左側のアイコンから「拡張機能」をクリック（四角が4つ並んだアイコン）
3. 検索ボックスに「Dev Containers」と入力
4. 「Dev Containers」（Microsoft製）を見つけてインストール

### 2-2. リポジトリをクローンする

フォークしたリポジトリを自分のパソコンにダウンロードします。

1. **フォークしたリポジトリのページを開く**
   - `https://github.com/あなたのユーザー名/rishu-app`

2. **Codeボタンをクリック**
   - 緑色の「Code」ボタンをクリック

3. **URLをコピー**
   - 「HTTPS」タブが選ばれていることを確認
   - URLをコピー（クリップボードアイコンをクリック）

4. **ターミナル（コマンドプロンプト）を開く**
   - **Windows**: スタートメニューから「コマンドプロンプト」または「PowerShell」を開く
   - **Mac**: 「アプリケーション」→「ユーティリティ」→「ターミナル」
   - **Linux**: ターミナルアプリを起動

5. **クローンするディレクトリに移動**
   ```bash
   cd クローンするディレクトリ
   # 例: cd ~/Desktop （デスクトップにクローンする場合）
   ```

6. **クローンコマンドを実行**
   ```bash
   git clone https://github.com/あなたのユーザー名/rishu-app.git
   ```
   - 「あなたのユーザー名」の部分は、実際のGitHubユーザー名に置き換えてください

7. **クローンされたディレクトリに移動**
   ```bash
   cd rishu-app
   ```

### 2-3. VS CodeでDev Containerを開く

1. **VS Codeでフォルダを開く**
   - VS Codeを起動
   - 「ファイル」→「フォルダを開く」
   - クローンした`rishu-app`フォルダを選択

2. **Dev Containerで開く**
   - ウィンドウ左下に青いボタン（「><」のようなアイコン）が表示されます
   - このボタンをクリック
   - メニューから「Reopen in Container」を選択
   - 開発したいサービスを選択（例: `frontend`、`syllabus-frontend`など）
   - 初回は環境構築に時間がかかります（通信環境によって数分〜30分程度）

3. **完了！**
   - 環境の構築が完了すると、コンテナ内でVS Codeが開きます
   - これで開発環境の準備が整いました

### 2-4. 元のリポジトリを追加する（upstream）

後で最新の変更を取り込めるように、元のリポジトリを登録します。

#### VS Codeで追加する方法（推奨）

1. **VS Codeのターミナルを開く**
   - メニューから「ターミナル」→「新しいターミナル」

2. **upstreamを追加**
   ```bash
   git remote add upstream https://github.com/ToYama170402/rishu-app.git
   ```

3. **確認**
   ```bash
   git remote -v
   ```
   - `origin`（あなたのフォーク）と`upstream`（元のリポジトリ）が表示されればOK

> **Note**: この操作は、VS Codeの拡張機能「Git Graph」などを使うと、より視覚的に管理できます。ただし、最初はターミナルで実行する方法を覚えておくと便利です。

## ステップ4: コードを変更する

いよいよコードを変更していきます。

### 4-1. 新しいブランチを作成する

メインブランチ（main）で直接作業せず、専用のブランチを作成します。

#### VS Codeでブランチを作成する（推奨）

1. **最新の状態を取得**
   - VS Codeの左下のブランチ名（「main」など）をクリック
   - 「main」を選択してメインブランチに切り替え
   - 「...」メニューから「プル」を選択して最新の状態を取得

2. **新しいブランチを作成**
   - 左下のブランチ名をクリック
   - 「新しいブランチの作成」を選択
   - ブランチ名を入力（例: `feature/add-graph-display`）
   
   **ブランチ名の規則:**
   - `feature/機能名` - 新機能の追加
   - `fix/バグ名` - バグ修正
   - `docs/ドキュメント名` - ドキュメントの更新
   - `refactor/対象` - リファクタリング
   - `test/テスト名` - テストの追加・修正
   - `chore/タスク名` - ビルド、設定、その他の雑多なタスク
   
   ブランチ名の例:
   - `feature/add-graph-display` （グラフ表示機能を追加）
   - `fix/login-bug` （ログインのバグを修正）
   - `docs/update-readme` （READMEを更新）
   
   詳しくは[CONTRIBUTING.md](./CONTRIBUTING.md#ブランチの作成)をご覧ください。

#### コマンドラインでブランチを作成する（参考）

```bash
# 最新の状態を取得
git checkout main
git pull upstream main

# 新しいブランチを作成
git checkout -b feature/あなたの機能名
```

### 4-2. コードを編集する

VS Codeでファイルを開いて編集します。

1. **ファイルを探す**
   - 左側のエクスプローラー（ファイル一覧）から編集したいファイルを探します
   - `Ctrl+P`（Mac: `Cmd+P`）でファイル名検索もできます

2. **編集する**
   - コードを変更します
   - わからないことがあれば、既存のコードを参考にしたり、質問したりしましょう

3. **保存する**
   - `Ctrl+S`（Mac: `Cmd+S`）で保存

### 4-3. 変更をテストする

変更が正しく動作するか確認します。

各サービスによってテスト・起動方法が異なります。詳しくは各サービスのREADMEをご覧ください：
- [frontend/README.md](./frontend/README.md)
- [syllabus-frontend/README.md](./syllabus-frontend/README.md)
- その他のサービスも同様に、各ディレクトリのREADMEを参照

ブラウザで表示を確認し、問題なく動作することを確認してください。

### 4-4. 変更をコミットする

変更をGitに記録します。VS Codeの画面から簡単に操作できます。

#### VS Codeでコミット・プッシュする（推奨）

1. **変更内容を確認**
   - VS Codeの左側のアイコンから「ソース管理」をクリック（分岐したアイコン）
   - 変更されたファイルの一覧が表示されます

2. **変更をステージに追加**
   - 変更ファイルの横にある「+」ボタンをクリック
   - すべての変更を追加する場合は、「Changes」の横にある「+」ボタンをクリック

3. **コミットメッセージを記入**
   - 上部のテキストボックスにコミットメッセージを入力
   - **コミットメッセージの形式**（[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/)に従う）:
     ```
     <type>: <subject>
     ```
   - **Type の種類**:
     - `feat`: 新機能
     - `fix`: バグ修正
     - `docs`: ドキュメントのみの変更
     - `style`: コードの動作に影響しない変更（空白、フォーマットなど）
     - `refactor`: バグ修正や機能追加ではないコードの変更
     - `test`: テストの追加や修正
     - `chore`: ビルドプロセスやツールの変更
   
   コミットメッセージの例:
   - `feat: グラフ表示機能を追加`
   - `fix: ログインできないバグを修正`
   - `docs: READMEに使い方を追加`
   
   詳しくは[CONTRIBUTING.md](./CONTRIBUTING.md#コミットメッセージ)をご覧ください。

4. **コミットする**
   - 「✓ コミット」ボタンをクリック

5. **変更をプッシュする**
   - 「...」メニューから「プッシュ」を選択
   - または、「ソース管理」パネル上部の「↑」アイコンをクリック
   - 初めてプッシュする場合、GitHubのユーザー名とパスワード（またはトークン）を聞かれることがあります

#### コマンドラインでコミット・プッシュする（参考）

VS Codeの操作が難しい場合は、コマンドラインでも実行できます：

```bash
# 変更内容を確認
git status

# 変更をステージに追加
git add .

# コミットする
git commit -m "feat: グラフ表示機能を追加"

# プッシュする
git push origin feature/あなたの機能名
```

## ステップ5: プルリクエストを作成する

変更を元のリポジトリに取り込んでもらうため、Pull Requestを作成します。

### 5-1. プルリクエストを作成

1. **GitHubのあなたのリポジトリを開く**
   - ブラウザで `https://github.com/あなたのユーザー名/rishu-app` にアクセス

2. **Compare & pull requestボタンをクリック**
   - プッシュした直後であれば、黄色いバナーに「Compare & pull request」ボタンが表示されます
   - 表示されない場合は、「Pull requests」タブから「New pull request」をクリック

3. **ベースとヘッドを確認**
   - **base repository**: `ToYama170402/rishu-app` base: `main` （取り込み先）
   - **head repository**: `あなたのユーザー名/rishu-app` compare: `feature/あなたの機能名` （あなたの変更）
   - 正しいか確認してください

4. **タイトルと説明を記入**
   - **タイトル**: 何を変更したか、簡潔に書く
     - 例: 「グラフ表示機能を追加」
   - **説明**: Pull Requestテンプレートに従って記入する
     - このリポジトリにはPull Requestテンプレートが用意されています
     - テンプレートの各項目を埋めることで、レビュワーが変更内容を理解しやすくなります
     - 何をしたか
     - なぜその変更が必要か
     - 関連するIssue（あれば）: `#123` のように書く
     - テストした内容

5. **Create pull requestをクリック**
   - これでPull Requestが作成されます

### 5-2. レビューを受ける

メンテナー（管理者）があなたのコードをレビューします。

- **コメントが付く**: 改善点や質問が来ることがあります
- **変更を求められる**: 追加の修正が必要な場合があります
- **承認される**: 問題なければ承認されます

### 5-3. 修正が必要な場合

レビューで修正を求められた場合：

1. **ローカルで修正**
   - VS Codeで該当箇所を修正

2. **コミット・プッシュ**
   ```bash
   git add .
   git commit -m "レビューの指摘を修正"
   git push origin feature/あなたの機能名
   ```

3. **自動的に反映される**
   - プッシュすると、Pull Requestに自動的に反映されます

### 5-4. マージされる

メンテナーが「Merge pull request」ボタンをクリックすると、あなたの変更が正式に取り込まれます。

**おめでとうございます！これであなたもrishu-appの貢献者です！🎉**

## 困ったときは

### よくある質問

#### Q1. コマンドがうまく動かない
- **A**: エラーメッセージをよく読んで、何が問題か確認しましょう。わからなければ、エラーメッセージをそのままIssueや質問で投稿してください。

#### Q2. Gitでエラーが出る
- **A**: Gitの基本的な使い方は[Git公式ドキュメント](https://git-scm.com/book/ja/v2)を参照するか、[GitHub Docs](https://docs.github.com/ja)を確認してください。

#### Q3. Pull Requestが取り込まれない
- **A**: メンテナーの都合で時間がかかることがあります。数日待っても反応がない場合は、優しくリマインドしても大丈夫です。

#### Q4. レビューで指摘された内容がわからない
- **A**: 遠慮なくPull Requestのコメント欄で質問してください。わからないことは恥ずかしいことではありません！
